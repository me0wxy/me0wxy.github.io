Java RMI & JNDI注入漏洞学习

发表于 2022-12-10

RMI

基本概念

RPC (Remote Procedure Call)，远程过程调用，像本地调用方法⼀样调用⼀个远程方法。

RMI (Remote Method Invocation)，Java远程⽅法调⽤，⼀种⽤于实现远程过程调⽤的应⽤程序编程接⼝，是RPC的一种具体实现，可以允许运行在一个Java虚拟机的对象调用运行在另一个Java虚拟机上的对象（的方法），实现了Java程序之间跨JVM的远程通信。

远程服务器实现具体的Java方法并提供接口，客户端本地仅需根据接口类的定义，提供相应的参数即可调用远程方法并获取执行结果，使得分布在不同的JVM中的对象的外表和行为都像本地对象一样。

工作原理

实现RMI意味着需要一个 Server 端和一个 Client 端。Server 端通常会创建一个对象，并使之可以被远程访问。 这个对象被称为远程对象。Server 端需要注册这个对象可以被 Client 远程访问。Client 端调用可以被远程访问的对象上的方法，Client 端就可以和 Server 端进行通信并相互传递信息。

从图中可以看到，Client 端有一个被称 Stub 的东西，有时也会被称为存根，它是 RMI Client 的代理对象，Stub 的主要功能是请求远程方法时构造一个信息块，RMI 协议会把这个信息块发送给 Server 端。这个信息块由几个部分组成：

  • 远程对象标识符
  • 调用的方法描述
  • 编组后的参数值（RMI协议中使用的是对象序列化）

既然 Client 端有一个 Stub 可以构造信息块发送给 Server 端，那么 RMI Server 端必定会有一个接收这个信息块的代理对象，称为 Skeleton。
它主要的工作是：

  • 解析信息块中的调用对象标识符和方法描述，在 Server 端调用具体的对象方法
  • 取得调用的返回值或者异常值
  • 把返回值进行编组，返回给客户端 Stub

此外还有一个比较重要的结构即RMI Registry：注册中⼼，类⽐成RMI的电话薄。类似⼀个⽹关，⾃⼰并不执⾏远程⽅法。但服务端可以在上⾯注册⼀个Name到对象的绑定关系。客户端通过这个Name向注册中⼼查询，得到这个绑定关系后，再链接服务端。使⽤注册中⼼查找对另⼀ 台主机上已经注册远程对象的引⽤。注册中⼼引导通过远程⽅法来传递远程引⽤

Server端在定义远程接口时，需要继承java.rmi.Remote接口，并且修饰符需要为public（否则远程调用的时候会报错），同时定义的方法里面需要抛出一个RemoteException异常；在实现远程接口的时候，需要继承 UnicastRemoteObject 类，实现其预定义的接口

JRMP & WebLogic T3

RMI是基于JRMP协议的，而Weblogic RMI是基于T3协议（也有基于CORBA的IIOP协议）。

JRMP TODO

WebLogic RMI是WebLogic对Java RMI的实现，它们之间的不同在于（1）WebLogic的字节码生成功能会自动生成服务端的字节码到内存。不再生成Skeleton骨架对象，也不需要使用UnicastRemoteObject对象（2）在WebLogic RMI 客户端中，字节码生成功能会自动为客户端生成代理对象，因此Stub也不再需要。
T3传输协议是WebLogic的自有协议，它有如下特点：（1）服务端可以持续追踪监控客户端是否存活（心跳机制），通常心跳的间隔为60秒，服务端在超过240秒未收到心跳即判定与客户端的连接丢失。（2）通过建立一次连接可以将全部数据包传输完成，优化了数据包大小和网络消耗。

RMI反序列化攻击

使用RMI进行反序列化攻击需要两个条件：调用的远程方法接收Object类型的参数、Server端存在执行命令利用链
（理解一下原因就是RMI在传输数据的时候会进行序列化，在传输完成后再进行反序列化。那么这时候如果传输一个恶意的序列化数据就会进行反序列化的命令执行）

RMI的核心之一是可以使⽤动态类加载和安全管理器来安全传输Java类，涉及到一个重要的属性java.rmi.server.codebase：该属性表⽰⼀个或多个URL位置，可以从中下载本地（ CLASSPATH ）找不到的类，相当于⼀个代码库。

对于客户端⽽⾔，如果服务端⽅法的返回值可能是⼀些⼦类的对象实例，⽽客户端并没有这些⼦类的class⽂件，如果需要客户端正确调⽤这些⼦类中被重写的⽅法，客户端就需要从服务端提供的 java.rmi.server.codebase URL去加载类；对于服务端⽽⾔，如果客户端传递的⽅法参数是远程对象接⼝⽅法参数类型的⼦类，那么服务端需要从客户端提供的 java.rmi.server.codebase URL去加载对应的类。
客户端与服务端两边的 java.rmi.server.codebase URL都是互相传递的。

⽆论是客户端还是服务端要远程加载类，都需要满足以下条件：

  1. 由于Java SecurityManager的限制，默认是不允许远程加载的，如果需要进⾏远程加载类，需要安装RMISecurityManager并配置java.security.policy
  2. .属性java.rmi.server.useCodebaseOnly的值必须为false

JNDI

基本概念

COBRA，LDAP

漏洞原理

CVE-2021-44228 (Log4j2 RCE)

基本原理：

https://xz.aliyun.com/t/6633#toc-7

将恶意的Reference类绑定在RMI注册表中，其中恶意引用指向远程恶意的class文件，当用户在JNDI客户端的lookup()函数参数外部可控或Reference类构造方法的classFactoryLocation参数外部可控时，会使用户的JNDI客户端访问RMI注册表中绑定的恶意Reference类，从而加载远程服务器上的恶意class文件在客户端本地执行，最终实现JNDI注入攻击导致远程代码执行

jndi注入的利用条件
客户端的lookup()方法的参数可控
服务端在使用Reference时，classFactoryLocation参数可控～
上面两个都是在编写程序时可能存在的脆弱点（任意一个满足就行），

漏洞修复与绕过

高版本修复与绕过：

除此之外，jdk版本在jndi注入中也起着至关重要的作用，而且不同的攻击响亮对jdk的版本要求也不一致，这里就全部列出来：

JDK 6u45、7u21之后：java.rmi.server.useCodebaseOnly的默认值被设置为true。当该值为true时，将禁用自动加载远程类文件，仅从CLASSPATH和当前JVM的java.rmi.server.codebase指定路径加载类文件。使用这个属性来防止客户端VM从其他Codebase地址上动态加载类，增加了RMI ClassLoader的安全性。

JDK 6u141、7u131、8u121之后：增加了com.sun.jndi.rmi.object.trustURLCodebase选项，默认为false，禁止RMI和CORBA协议使用远程codebase的选项，因此RMI和CORBA在以上的JDK版本上已经无法触发该漏洞，但依然可以通过指定URI为LDAP协议来进行JNDI注入攻击。

JDK 6u211、7u201、8u191之后：增加了com.sun.jndi.ldap.object.trustURLCodebase选项，默认为false，禁止LDAP协议使用远程codebase的选项，把LDAP协议的攻击途径也给禁了。
